# 中文 API 注释规范

本文档用于统一本项目公共接口与关键入口函数的中文注释风格，重点解决“接口职责不清、线程边界难读、数据流转不直观”的问题。

## 目标

- 让公共 API 在不跳实现的前提下可快速理解职责
- 让线程边界、回调上下文、所有权语义一眼可见
- 让播放链路中的状态变化、时钟基准、队列语义更容易追踪
- 保持注释可维护，避免把注释写成重复代码的噪声

## 适用范围

### 优先级 1：`include/*`

优先为以下对象补 `///` 注释：

- 对外暴露的类、结构体、枚举
- 公共方法
- 回调类型
- 生命周期接口（`open` / `close` / `init` / `stop`）
- 线程/队列/时钟相关接口

### 优先级 2：`src/*`

实现文件不追求“每行都注释”，只给关键入口补注释：

- 播放链路入口函数
- 渲染线程 / 解码线程 / 音频回调线程入口
- 关键状态切换函数
- 复杂资源初始化与释放逻辑
- 容易误解的数据复制、所有权转移、flush 逻辑

## 注释层级

### `///`

用于 API 级注释，放在声明或定义前，适用于：

- 类 / 结构体 / 枚举
- 公共函数
- 线程入口函数
- 回调函数

示例：

```cpp
/// 打开媒体文件并初始化解复用、解码和渲染资源。
bool open(const std::string& filename);
```

### `//`

用于实现内的局部说明，适用于：

- 一段实现为何这样写
- 锁顺序/线程竞争规避
- 某个边界条件的原因
- 平台兼容性差异

示例：

```cpp
// 避免与 SDL 音频回调线程形成设备锁和队列锁反转。
SDL_PauseAudioDevice(audio_device_, 0);
```

## 必写内容

对公共 API，至少说明下面四类信息中的一部分；如果缺失会影响理解，就必须写出来。

### 1. 做什么

先写职责，不先写实现。

- 好：`/// 消费一次下一章节请求。`
- 差：`/// 将 previous_chapter_requested_ 设为 false。`

### 2. 参数/返回值语义

当参数不是“看名字就完全明白”时，要补语义。

- 时间单位是否为秒
- 比例是否在 `[0, 1]`
- 指针是否转移所有权
- 返回 `false` 的主要语义是什么

建议写法：

```cpp
/// 按秒 seek 到目标时间；成功后会清除 EOF 状态。
bool seek(double timestamp);
```

### 3. 线程/回调上下文

只要接口可能跨线程，必须写。

- 谁调用
- 真正执行发生在哪个线程
- 是否允许阻塞

建议写法：

```cpp
/// SDL 音频回调线程入口；从队列拉取数据并推进播放时间戳。
/// @note 回调线程必须保持低延迟，不能做阻塞操作。
static void audioCallback(void* userdata, uint8_t* stream, int len);
```

### 4. 副作用/状态变化

以下情况要写清楚：

- 是否清队列 / flush
- 是否重置 EOF
- 是否触发后台线程行为
- 是否只记录请求，不立即执行

建议写法：

```cpp
/// 异步切换全屏状态，真正切换发生在渲染线程。
void toggleFullscreen();
```

## 推荐写法

### 单行职责优先

优先使用“做什么 + 约束”两行以内的短注释。

```cpp
/// 返回音频设备实际播放到的时间戳，通常可作为主时钟。
double getPlaybackPts() const;
```

### 参数复杂时再加标签

只在必要时使用 `@param`、`@return`、`@note`：

```cpp
/// 追加一段 PCM 数据到播放队列。
/// @param pts 该音频块起始时间戳；小于 0 时按队列尾部自动推算。
void play(const std::vector<uint8_t>& data, double pts = -1.0);
```

### 先写外部语义，再写内部约束

```cpp
/// 向渲染线程提交一帧视频数据。
/// @note 当前实现约定 `data` 指向 `AVFrame`，调用线程只排队不直接显示。
void renderFrame(const uint8_t* data, int width, int height);
```

## 禁止项

不要写下面这些注释：

- 单纯复述代码字面行为
- 与实现明显脱节的历史说明
- 大段情绪化、评价式文字
- 不说明约束的空泛描述

反例：

```cpp
/// 这是一个非常重要的函数。
/// 这里做了一些处理。
```

## 与本项目结合的重点

本项目是播放器链路，以下位置必须优先注释：

- `PlayerCore`：播放生命周期、线程启动/停止、状态切换
- `Scheduler`：时钟驱动、丢帧、等待、渲染节拍
- `Display`：事件线程/渲染线程边界、请求消费模型
- `AudioPlayer`：SDL 回调线程、音频时钟来源、缓冲语义
- `Demuxer`：读包、seek、EOF、章节与流探测
- `FrameQueue` / `ThreadSafeQueue`：生产者-消费者语义、flush/stop/EOF 行为

## 推荐执行顺序

每次补注释按下面顺序做：

1. 先看头文件，标出公共入口
2. 再看对应实现，确认线程和副作用
3. 先补类注释，再补生命周期函数
4. 再补线程入口、回调入口、队列接口
5. 最后补少量实现内 `//` 说明复杂块

## 验收清单

补完注释后，至少自检以下问题：

- 是否说明了“做什么”而不是“代码怎么写”
- 是否写清楚时间单位、范围、返回值语义
- 是否标出线程边界和回调上下文
- 是否写清楚所有权、flush、重置、异步执行等副作用
- 是否避免了无价值重复描述
- 注释是否可以在不看实现时帮助理解 API

## 项目内统一约定

- 公共接口优先使用 `///`
- 实现细节说明使用 `//`
- 中文为主，必要的技术词保留英文原词，如 `flush`、`callback`、`renderer`
- 注释尽量短句，避免超过 3 行
- 先写“做什么”，再写“限制/副作用/线程上下文”

## 参考风格来源

本规范吸收以下常见风格的优点：

- C++/Doxygen 的三斜杠接口文档风格
- 播放器项目中常见的“线程边界优先”注释方式
- 渲染/音频模块中常见的“职责 + 约束 + 副作用”短注释方式
