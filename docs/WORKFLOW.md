# 工作流程规范

本文档记录开发过程中的工作流程规范。

---

## 代码提交规范

### 提交时机

**不要直接提交代码到仓库**。每次代码修改后，应该：

1. **先展示修改内容**：向用户展示改动的文件和内容
2. **等待用户确认**：询问用户是否需要提交
3. **用户确认后**：再执行 git add + commit + push

### 提交信息格式

使用简洁的中文描述：

```
fix: 修复视频流索引不匹配问题

docs: 更新文档，添加问题修复记录
```

---

## 文档更新规范

### 文档分类

```
docs/
├── README.md          # 文档索引（入口）
├── CHANGELOG.md      # 问题修复记录（重要）
├── VERSION.md        # 版本记录与依赖说明
├── ARCHITECTURE.md   # 架构设计
├── IMPLEMENTATION.md # 实现步骤
├── WINDOWS_SETUP.md  # Windows 配置
└── LOGGING.md        # 日志系统说明
```

### 文档同步要求

每次修复问题时，必须同步更新以下文档：

| 修复类型 | 需要更新的文档 |
|----------|---------------|
| Bug 修复 | CHANGELOG.md + VERSION.md |
| 新增功能 | VERSION.md + 相关功能文档 |
| 文档更新 | 文档更新日志 |
| 依赖变更 | VERSION.md + WINDOWS_SETUP.md |

### CHANGELOG.md 结构

每个问题修复应包含：

```markdown
## 问题 N: 问题标题

**日期**: YYYY-MM-DD

### 问题描述
- 问题现象描述

### 原因分析
- 为什么会发生这个问题

### 解决方案
- 如何修复的

### 修改文件
- 列出所有修改的文件
```

## 问题排查流程

### 调试步骤

1. **分析日志**：根据日志输出定位问题
2. **定位代码**：找到问题所在的源文件
3. **分析原因**：理解问题的根本原因
4. **提出方案**：给出修复方案
5. **等待确认**：让用户确认后再修改

### 调试日志启用

```bash
cmake -B build -DDEBUG_MODE=ON
cmake --build build
```

## 工作流程示例

### 正确流程

```
用户: 视频无法播放

我: (分析问题) 发现是流索引问题
我: 向您解释问题原因和解决方案
我: 等待您确认

用户: 好的，修复吧

我: (修改代码)
我: 编译测试
我: 向您展示修改内容
我: 询问是否需要提交

用户: 提交吧

我: git add + commit + push
```

### 不应该的做法

- 直接修改代码后立即提交
- 不向用户解释就直接推送
- 忽略文档同步

---

## 注意事项

1. **不主动提交**：除非用户明确要求，否则不执行 git push
2. **解释先行**：修改前先解释问题和方案
3. **文档同步**：每次修复都要同步更新相关文档
4. **测试验证**：修改后先本地测试再提交
